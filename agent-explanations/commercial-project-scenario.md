# 商业性项目完整场景与 Prompt 讲解

[返回总目录](./README.md)

## 场景设定

以一个典型的商业型双创项目为例：团队计划做一款面向高校大学生的 AI 求职陪练平台，核心价值主张是“用更低成本、更高频次的模拟训练，提高学生求职面试通过率”。这类项目的关键不只是“产品能不能做出来”，还包括“用户是否真实存在、需求是否足够强、是否愿意付费、获客路径是否成立、单位经济能否闭环”。

因此，在这一类场景中，智能体不会只把输入当成一段创意介绍，而是会把它放入一条更完整的创业推理链中进行理解：

**用户痛点 -> 解决方案 -> 差异化优势 -> 商业模式 -> 渠道增长 -> 财务合理性 -> 执行路径**

## 系统如何理解这个场景

在商业性项目里，智能体的目标不是直接替团队写出一份“看起来很完整”的商业计划书，而是帮助团队把一个“有想法的项目”逐步推进为一个“有商业成立可能的项目”。

因此，系统在这一场景下会优先关注以下几个判断：

1. 目标用户是否足够清晰，是否存在“面向所有人”的空泛表达。
2. 痛点是否有真实证据支撑，而不是团队主观想象。
3. 解决方案相对于现有替代方案是否真的更优。
4. 商业模式是否成立，包括谁付费、为什么付费、如何持续付费。
5. 获客渠道是否可达，是否存在“全靠自然增长”的不现实假设。
6. 财务逻辑是否初步说得通，例如 CAC、LTV、成本结构和盈亏平衡点。

从系统设计视角看，这类判断本质上是在检验项目是否形成了一个完整的商业闭环，而不是只检验“创意是否新颖”。

## Prompt 设计思路

商业项目的 Prompt 设计重点，不是让模型生成一段表面流畅的计划书文字，而是要求模型沿着“商业闭环”逐层检查项目是否成立。整体上可以分为三层。

### 1. 身份定位

首先明确模型的角色，例如“创新创业项目教练”“商业计划顾问”或“评审视角分析者”。这样模型的输出重点会从泛化问答转向创业判断。

### 2. 分析维度约束

要求模型围绕以下维度展开分析：

- 用户画像与使用场景
- 核心痛点与证据
- 产品方案与差异化
- 收费对象与变现逻辑
- 获客渠道与转化路径
- 财务可行性与关键风险
- 下一步最小验证动作

这一层的作用，是避免模型只停留在“创意不错”“方向很好”之类的空泛评价上，而是真正去判断项目能否推进。

### 3. 输出结构约束

要求输出必须包含以下内容：

- 当前最大瓶颈
- 2 到 3 个苏格拉底式追问
- 1 个下一步唯一任务
- 明确的验收标准

这样设计的原因是，双创项目最怕“建议很多，但没有动作”。系统最终是为了推动团队继续验证和迭代，而不是提供一堆无法执行的泛化建议。

## 典型 Prompt 示例

```text
你是一名创新创业项目教练，请从商业成立性的角度分析该项目。
不要直接代写计划书，而是先判断：
1. 目标用户是否清晰；
2. 痛点是否有证据；
3. 解决方案是否相对现有替代方案有明显改进；
4. 商业模式是否成立；
5. 获客渠道是否可执行；
6. 财务逻辑是否初步合理。

请输出：
- 当前最关键的瓶颈
- 3个苏格拉底式追问
- 1个下一步唯一任务
- 该任务的验收标准
```

## 这种 Prompt 为什么有效

因为商业项目的核心，不是把故事讲得足够宏大，而是证明存在一个可持续的价值交换关系。对应到系统输出上，这类 Prompt 会天然强调三个关键词：

- **聚焦**：先收缩到核心用户和核心场景。
- **验证**：先证明需求、支付意愿和渠道可达性。
- **闭环**：最终把用户价值、收入模式、成本结构和增长路径连接起来。

因此，这一类 Prompt 实际上对应的是一种“从创意走向商业成立”的推进逻辑。

## 可写入最终文档的总结

在商业性项目场景下，智能体更强调项目是否形成了“用户价值 - 商业模式 - 增长路径 - 财务逻辑”的商业闭环。系统并不鼓励团队停留在创意描述层面，而是通过结构化 Prompt 持续追问项目是否有真实痛点、是否可触达用户、是否具备支付可能、是否形成基本单位经济模型。由此，模型输出的不只是建议文本，更是一种推动项目从“想法”走向“成立”的过程化辅导。
