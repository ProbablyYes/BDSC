# 第一部分：多智能体编排

## 1. 什么叫"多智能体"

我们的系统里有 **6 个独立的分析 Agent（智能体）**，各有不同角色：

| Agent | 中文名 | 职责 |
|-------|--------|------|
| **coach** | 项目教练 | 判断项目阶段，抓核心瓶颈，启发式追问 |
| **analyst** | 风险分析师 | 挖掘结构层原因，识别反直觉盲区，分析风险模式 |
| **advisor** | 竞赛顾问 | 评委视角，评分区间，证据链强弱 |
| **tutor** | 课程导师 | 概念讲解，方法论桥接，教学性内容 |
| **grader** | 评分官 | 量规打分，维度覆盖评估 |
| **planner** | 行动规划师 | 拆解下一步任务，验收标准 |

另有 **1 个编排器（orchestrator）**，不做独立分析，只负责把上面 6 个 Agent 的输出**组织成一条连贯的自然语言回复**。

所以一共是 **6 + 1 = 7 个角色**参与一轮对话。

## 2. 什么叫"编排"——原理

"编排"要回答四个问题：**本轮调谁？谁先谁后？每个人看到什么信息？最后怎么合成？**

### 2.1 本轮调谁：维度激活机制

传统做法是查一张"意图×复杂度"的矩阵，得到固定的 Agent 列表。但这导致同一类问题永远调同样的 Agent，不管学生材料是三句话还是三页纸。

我们的做法：把回复拆成 **10 个分析维度**（相当于"评委脑子里的思考切面"），每个维度独立决定要不要激活。

**10 个分析维度：**

| # | 维度 | 含义 | 是否必选 |
|---|------|------|---------|
| 1 | 项目状态判断 | 项目处于什么阶段，逻辑通不通 | ✅ 必选 |
| 2 | 核心瓶颈识别 | 最制约推进的 1-2 个瓶颈 | ✅ 必选 |
| 3 | 结构层原因 | 表面问题背后共同指向的深层断点 | ❌ 按需 |
| 4 | 反直觉洞察 | 被忽略的盲区、过于乐观的假设 | ❌ 按需 |
| 5 | 方法论桥接 | 讲清一个概念并桥接回项目 | ❌ 按需 |
| 6 | 评委判断标准 | 评审者视角的得分区间 | ❌ 按需 |
| 7 | 外部案例/数据 | 竞品、行业数据、案例对比 | ❌ 按需 |
| 8 | 策略方向 | 可选打法与取舍 | ❌ 按需 |
| 9 | 行动方案 | 本周该做什么、验收标准 | ❌ 按需 |
| 10 | 启发式追问 | 推动学生思考的追问 | ✅ 必选 |

**激活公式：** 每个维度算三个分量，相乘得到激活分数：

```
激活分数 = relevance（相关性）× uncertainty（不确定性）× impact（影响力）
```

- **relevance**：这个维度与当前意图的关联度。例如"项目诊断"意图下"核心瓶颈"的 relevance = 0.95，但"闲聊"意图下只有 0.1。系统内置了一张 10×10 的查找表。
- **uncertainty**：学生消息中对这个维度的模糊程度。如果学生已经明确写了竞品分析，"反直觉洞察"的 uncertainty 就低；反之如果完全没提竞品，uncertainty 就高。
- **impact**：这个维度对最终判断的重要性。例如竞赛模式下"评委判断标准"的 impact 更高。

激活阈值：**分数 > 0.25** 或者维度被标记为"必选"→ 激活。

激活后还有一轮**精炼（refine）**：
- 如果超图覆盖度 < 4/10 → 强制激活"结构层原因"
- 如果诊断触发规则 ≥ 4 条 → 提升"反直觉洞察"的分数
- 如果 RAG 检索到高相似案例 → 提升"外部案例/数据"的分数

这样同一类问题，材料丰富时可能激活 7-8 个维度，材料简单时可能只激活 3 个，**回复的深度和广度自动适配**。

### 2.2 谁负责写哪个维度：维度所有权（DIM_OWNERSHIP）

每个维度指定一个**主笔（Writer）**和可选的**挑战者（Challenger）**：

| 维度 | 主笔 | 挑战者 | 为什么这样分 |
|------|------|--------|-------------|
| 项目状态判断 | Coach | — | 教练最适合做整体判断 |
| 核心瓶颈识别 | Coach | **Analyst** | 教练找瓶颈，分析师挑刺 |
| 结构层原因 | Analyst | — | 分析师擅长结构性思考 |
| 反直觉洞察 | Analyst | — | 需要逻辑审计能力 |
| 方法论桥接 | Tutor | — | 导师擅长教概念 |
| 评委判断标准 | Advisor | **Grader** | 顾问给判断，评分官验证 |
| 外部案例/数据 | Advisor | **Tutor** | 顾问找外部案例，导师补充视角 |
| 策略方向 | Coach | — | 教练给方向 |
| 行动方案 | Planner | — | 规划师专门做任务拆解 |
| 启发式追问 | Orchestrator | — | 编排器在最终合成时生成 |

**挑战者的作用：** 主笔写完后，挑战者会收到主笔的结论，用另一个 LLM 调用专门找"最站不住脚的点"、"被遗漏的考虑"和"替代解释"。如果挑战者发现实质性问题，该维度会被标记为"有争议"，编排器会呈现双方观点而非一家之言。

### 2.3 谁先谁后：条件依赖 + 拓扑排序

有些维度必须建立在其他维度的结论之上。例如"策略方向"要参考"核心瓶颈"的判断，不能凭空给策略。

我们用**带权重的条件依赖**定义这些关系：

```
策略方向 依赖:
  - 核心瓶颈（权重 1.0，必须等它完成）
  - 结构层原因（权重 0.7，仅当复杂度 ≥ 3 时才等）

行动方案 依赖:
  - 核心瓶颈（1.0）
  - 策略方向（0.9）

评委判断标准 依赖:
  - 项目状态判断（0.8）
  - 外部案例/数据（0.5，仅竞赛模式）
```

系统对所有激活维度做**拓扑排序**，自动划分执行阶段：
- Phase 1：没有依赖的 Agent 先跑（coach、tutor 等）
- Phase 2：依赖 Phase 1 结果的 Agent 跑（analyst 等）
- Phase 3：依赖 Phase 2 的 Agent 跑（planner 等）

**不是固定阶段数，而是根据本轮激活的维度和依赖关系动态生成的。**

### 2.4 最后怎么合成：编排器 + 回复策略

所有 Agent 的输出汇总后，编排器（Orchestrator）负责合成。它的权限受到严格约束：

**编排器可以做的：**
- 决定段落顺序和详略比例
- 如果某维度标注"有争议"，呈现双方观点
- 如果某维度标注"低置信"，措辞上体现不确定

**编排器不能做的：**
- 不能引入分析结论中不存在的新事实
- 不能推翻任何 Writer 的主结论
- 不能提到内部角色名称
- 不能编造竞品名称、市场数字

同时，系统根据问题复杂度自动选择 **8 种回复策略** 之一来指导编排器：

| 策略 | 适用场景 | 长度范围 |
|------|---------|---------|
| **comprehensive** | 复杂查询 + 项目类 intent + 3+ 维度 | 2500-4500字 |
| **panorama** | 中等复杂 + 5+ 维度 | 2000-3800字 |
| **deep_dive** | 聚焦单点分析 | 1200-2500字 |
| **challenge** | 有争议维度 / 反直觉洞察 | 800-1600字 |
| **progressive** | 探索阶段 / 学生想法还不成熟 | 400-800字 |
| **teach_concept** | 概念学习 | 700-1200字 |
| **compare** | 竞品/市场分析 | 1000-2000字 |
| **casual** | 闲聊 | 100-350字 |

策略选择取决于：消息复杂度评分（文本长度、富内容信号词、结构化标记）、诊断丰富度（KG 实体数、规则数、缺口数）、意图类型、和维度激活数量。

# 第二部分：超图设计

## 1. 为什么要超图（和普通图的区别）

**普通图**（Neo4j 里那张）：一条边只连接两个节点，例如 `Project --HAS_PAIN--> PainPoint`。它适合存储**事实关系**。

**超图**：一条"超边"可以同时连接**多个节点**。例如一条超边同时连接"用户A"+"痛点B"+"方案C"+"商业模式D"，表示"这四个东西要一起成立才是一个完整的商业闭环"。

我们用超图解决的核心问题是：**不只检查"你有没有写某个维度"，而是检查"你写的这些维度之间有没有形成评委关心的逻辑闭环"。**

## 2. 超图的三层设计

### 第一层：15 个超边家族（Edge Families）

这是超图最核心的设计——从案例库中挖掘出 **15 种类型的超边**，每种代表一类跨维度的分析视角：

| # | 超边家族 | 中文名 | 目标生成数量 | 语义 |
|---|---------|--------|------------|------|
| 1 | Value_Loop_Edge | 价值闭环超边 | 16 | 用户→痛点→方案→商业模式是否闭合 |
| 2 | User_Pain_Fit_Edge | 用户痛点匹配超边 | 12 | 目标用户与痛点是否对得上 |
| 3 | Risk_Pattern_Edge | 风险模式超边 | 18 | 历史案例中反复出现的风险模式 |
| 4 | Evidence_Grounding_Edge | 证据锚定超边 | 12 | 关键假设是否有证据支撑 |
| 5 | Market_Competition_Edge | 市场竞争超边 | 10 | 市场定位与竞争格局是否合理 |
| 6 | Execution_Gap_Edge | 执行断裂超边 | 10 | 从方案到落地之间有没有断层 |
| 7 | Compliance_Safety_Edge | 合规安全超边 | 8 | 涉及敏感场景时合规措施是否到位 |
| 8 | Innovation_Validation_Edge | 创新验证超边 | 8 | 创新点是否经过验证 |
| 9 | Pricing_Unit_Economics_Edge | 定价单元经济超边 | 10 | 定价逻辑与单位经济是否可持续 |
| 10 | Substitute_Migration_Edge | 替代迁移超边 | 10 | 用户从替代方案迁移过来的成本与动力 |
| 11 | Trust_Adoption_Edge | 信任采纳超边 | 8 | 用户信任建立与采纳路径 |
| 12 | Retention_Workflow_Embed_Edge | 工作流嵌入超边 | 8 | 产品是否能嵌入用户工作流以保留存 |
| 13 | Rule_Rubric_Tension_Edge | 规则评分张力超边 | 10 | 风险规则与量规评分之间的冲突与张力 |
| 14 | Stage_Goal_Fit_Edge | 阶段目标匹配超边 | 8 | 当前阶段的目标是否合理 |
| 15 | Ontology_Grounded_Edge | 本体落地超边 | 6 | 概念是否与本体概念节点对齐 |

**总计目标生成量**：最多约 **154 条**超边（实际数量取决于案例库中有多少项目能产生有效的跨维度模式）。

**建库过程（`rebuild` 函数）：**
1. 从 Neo4j 加载所有案例项目的完整维度数据
2. 对每个项目，遍历上述 15 种家族的生成逻辑，把满足条件的维度组合产生超边
3. 按支持度（support，即多少个项目产生了同一模式）排序
4. 每个家族取 top-N 条，用 HyperNetX 库建超图数据结构
5. 持久化到 Neo4j（`HyperNode`、`Hyperedge` 节点）

### 第二层：20 个闭环模板（T1–T20）

模板用于**检测学生项目的逻辑闭环完整性**。每个模板定义"哪几个维度要同时出现"：

| 模板 | 名称 | 涉及维度 | 语义 |
|------|------|---------|------|
| T1 | 痛点-人群-方案-商业模式闭环 | stakeholder + pain_point + solution + business_model | 核心商业故事线 |
| T2 | 痛点证据链 | stakeholder + pain_point + evidence | 痛点有没有真实验证 |
| T3 | 方案-技术-市场匹配 | solution + technology + market | 技术路线和目标市场对不对 |
| T4 | 市场-竞品-方案定位 | market + competitor + solution | 竞争格局中方案定位清不清 |
| T5 | 商业模式-财务逻辑 | business_model + resource + team | 商业模式是否可执行 |
| T6 | 单位经济模型闭环 | business_model + market + resource | 单位经济能否持续 |
| T7 | 增长路径 | stakeholder + channel + market | 获客渠道与市场匹配 |
| T8 | 团队-里程碑-资源 | team + resource + market | 团队能力支撑里程碑 |
| T9 | 风险与合规 | risk + evidence + market | 高风险场景有没有控制 |
| T10 | 问题-方案-证据三角 | stakeholder + pain_point + solution + evidence | 最核心的三角验证 |
| T11 | 市场规模-增长假设 | market + business_model | 市场假设和增长路径一致吗 |
| T12 | 竞品-护城河 | competitor + resource + business_model | 护城河和资源壁垒 |
| T13 | 技术可行性 | technology + team + resource | 技术与团队资源匹配 |
| T14 | 证据-财务 | evidence + business_model | 财务参数有数据支撑 |
| T15 | 证据-市场规模 | evidence + market | 市场规模有公开数据 |
| T16 | 用户-渠道匹配 | stakeholder + channel | 渠道和用户行为匹配 |
| T17 | 痛点-方案-定价 | pain_point + solution + business_model | 定价与痛点强度一致 |
| T18 | 团队-风险 | team + risk | 关键风险有负责人 |
| T19 | 完整商业闭环 | stakeholder + pain_point + solution + market + competitor + business_model | **最完整的 6 维闭环** |
| T20 | 增长-护城河 | market + business_model + resource | 增长与壁垒互相支撑 |

系统会对每个模板输出三种状态：
- `complete`：所有涉及维度都在学生材料中出现 → 该闭环完整
- `partial`：部分维度缺失 → 闭环有缺口
- `missing`：全部维度都缺 → 该视角完全没覆盖

### 第三层：20 条一致性规则（G1–G20）

即使学生"写了某些维度"，逻辑也可能不成立。一致性规则检查的是**维度之间的逻辑一致性**：

| 规则 | 触发条件 | 追问示例 |
|------|---------|---------|
| G1 | 完全没有竞品或替代品 | "请列出至少3个用户当前可能在用的替代方案" |
| G2 | 有用户和痛点但没有任何证据 | "你是否进行过至少8次深度访谈？" |
| G3 | 写了方案但没有明确痛点 | "请用一句话说清楚哪类用户在什么场景被什么折磨" |
| G4 | 提到CAC但没有渠道转化逻辑 | "你的获客渠道具体是什么？每个渠道转化率假设？" |
| G5 | 出现"1%市场份额"式跳跃假设 | "请自下而上估算第一年你能获得多少付费用户" |
| G6 | 缺少市场维度 | "请给出目标市场的大致规模与数据来源" |
| G7 | 有方案但没有商业模式 | "谁在什么场景下，为你的方案实际掏钱？" |
| G8 | 谈到营收/盈利但无数据来源 | "你的收入与成本假设来自哪些数据或实验？" |
| G9 | 涉及隐私/医疗等高风险但无控制 | "请画出数据流图并标出合规措施" |
| G10 | 技术复杂但没有团队说明 | "谁来负责核心技术？他的相关经验是什么？" |
| G11 | 只有模糊的"线上推广" | "请列出3个具体渠道并估算 CAC" |
| G12 | 技术堆叠过多而痛点不明确 | "请删掉80%技术描述，只保留与核心痛点相关的" |
| G13 | 声称有护城河但无竞品对比 | "请完成至少3个竞品/替代品的对比表" |
| G14 | 没有从0到1的增长路径 | "用3-5步描述从第一个付费用户到100个的路径" |
| G15 | 依赖关键资源但未说明获取方式 | "这些资源在谁手里？如何以可接受成本获得？" |
| G16 | 有团队但缺少里程碑规划 | "未来3个月各自要完成哪些可验收的里程碑？" |
| G17 | 有零散数据但未形成证据链 | "请按用户-痛点-方案效果三维度重新整理证据" |
| G18 | "颠覆/革命性"等强表述但无指标 | "请给出3个可量化指标证明你的颠覆性" |
| G19 | 没有明确的北极星指标 | "如果只选一个指标衡量项目成败，你会选什么？" |
| G20 | 完全没有风险章节 | "请列出3个最可能让项目失败的风险及缓解措施" |

每条规则触发后会生成**追问句**（pressure questions），供教练和分析师 Agent 在回复中使用。

## 3. 超图如何参与智能体决策

超图不直接替学生写答案，它做三件事：

**（1）控制 Gather 的轻重（分层按需）**

系统的数据收集分两层：
- Layer 0（必跑）：RAG 向量检索 + KG 知识抽取 + 条件性 Web 搜索
- Layer 1（按需）：超图教学库分析 + 超图学生分析 + Neo4j 图遍历

Layer 1 是否执行取决于"数据成熟度"：
- **cold**（冷启动）：实体 < 6 且规则 < 2 → 不跑 Layer 1
- **warm**（已有积累）：实体 ≥ 6 或规则 ≥ 2 → 跑 Layer 1
- **hot**（数据丰富）：实体 ≥ 12 且规则 ≥ 4 → 全力跑 Layer 1

**（2）影响维度激活**

超图覆盖度评分直接影响编排决策：
- 覆盖度 < 4/10 → 强制激活"结构层原因"维度（让分析更关注骨架缺失）
- 覆盖度 ≥ 8/10 → 回复更倾向"锦上添花"而非"基础补课"

**（3）按角色裁剪信号注入（`_fmt_hyper_for_agent`）**

同一份超图分析结果，不同 Agent 只看到与自己职能相关的部分：

| Agent | 收到的超图信号 |
|-------|--------------|
| **Coach** | 维度覆盖度（如 6/10）、未闭合的逻辑环（如 T1 缺 business_model）、一致性问题 + 追问句、价值链路完整度、拓扑洞察 |
| **Analyst** | 风险模式警告（pattern_warnings）、特定 G 规则（G1/G3/G5/G8/G9/G12/G18/G20）、教学超边、关键缺失维度、断裂链路 |
| **Advisor** | 闭环完成率（如 8/20）、覆盖度是否达竞赛基准（8/10）、关键模板 T1/T10/T19 的缺失情况 |
| **Grader** | 覆盖度总览、一致性问题总数、闭环完成率 |
| **Planner** | 待补维度 + 推荐行动、行动线索、可推进的未闭合模板 |
| **Tutor** | 学生维度覆盖度（简化版） |

这样每个 Agent 的 prompt 里只有与它分工相关的超图片段，**避免信息过载也避免互相干扰**。

# 第三部分：知识图谱（Neo4j）

## 1. 图数据库 Schema

Neo4j 中存储了案例库的完整结构化数据：

**16 种节点标签：**
- Project（项目根节点）、Category（类别）
- Stakeholder（目标用户）、PainPoint（痛点）、Solution（方案）
- InnovationPoint（创新点）、BusinessModelAspect（商业模式）
- Market（市场）、ExecutionStep（执行步骤）、RiskControlPoint（风控）
- Evidence（证据）、RiskRule（风险规则）、RubricItem（量规项）
- OntologyNode（本体概念）、EducationLevel（教育层次）、AwardLevel（奖项）

**15 种关系类型：**
- `BELONGS_TO`（属于类目）、`HAS_TARGET_USER`、`HAS_PAIN`、`HAS_SOLUTION`
- `HAS_INNOVATION`、`HAS_BUSINESS_MODEL`、`HAS_MARKET_ANALYSIS`
- `HAS_EXECUTION_STEP`、`HAS_RISK_CONTROL`、`HAS_EVIDENCE`
- `HITS_RULE`（触发风险规则）、`EVALUATED_BY`（量规评估）
- `INSTANCE_OF`（概念归属）、`TARGETS_CONCEPT`、`REQUIRES_EVIDENCE_FROM`

## 2. 双通道检索：向量 + 图遍历

| 通道 | 技术 | 回答的问题 |
|------|------|-----------|
| 向量语义检索（RAG） | bge-m3 embedding + 余弦相似度 | "整体上哪个案例和我最像？" |
| Neo4j 图遍历 | Cypher 查询 + 三策略 | "结构上谁和我面临同一痛点？谁在我缺的维度上更完整？" |

**Neo4j 图遍历的三个策略：**

1. **共享节点扇出**：找到与学生项目共享同一个 PainPoint / Solution / BusinessModel 等节点的其他项目 → "谁也面对同样的问题"
2. **结构互补匹配**：学生有痛点但没商业模式 → 找同时有两者的项目 → "谁在你缺失的维度上做得完整"
3. **关键词回退**：前两个策略召回不足时，按实体名关键词模糊匹配扩大范围

命中后为每个项目拉取完整上下文（痛点、方案、创新、商业模式、证据），前端展示"别人怎么做的"对比卡片。
