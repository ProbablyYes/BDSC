# 知识图谱及评估说明

本文对应学生端顶部栏中的`知识图谱 / 质量评估`面板。
- 大框架是什么
- 每个维度对应老师到底在担心什么
- 系统分别用了哪些指标去回答这些问题
- 当前实际结果如何，应如何解释

这篇文档的核心目标，是把“知识图谱质量评估”从一个分数，讲成一套**可追问、可复核、可解释**的方法体系。

---

## 一、评估总框架

你给出的评估框架可以整理成下面这张总表。它不是简单列指标，而是把“老师关心什么”与“系统用什么证明”对齐。

| 评估维度 | 老师真正关心的问题 | 系统对应指标 | 当前结论 |
| --- | --- | --- | --- |
| A. 抽取正确性 Correctness | 抽出来的是不是真的？有没有抽错类、漏关键词？ | `抽取准确率`、`语义有效性`、抽样审计 | 整体较强，但“语义有效性”仍是当前最弱项 |
| B. 抽取完整性 Completeness | 该有的都有吗？高频概念是不是高频？低频是不是合理低频？ | `知识覆盖率`、`维度覆盖率`、`缺失控制`、维度分布 | 总体完整，`商业模式`是最值得继续补强的通用维度 |
| C. 证据可追溯 Traceability | 每条抽出来的东西能不能回到原文？有没有 `quote`？ | `证据可溯源性`、`traceability_rate`、`audit_closure` | 很强，是当前系统最稳定的优势之一 |
| D. 结构合理性 Structural Coherence | 图组织得有没有内在规律？是否符合本体约束？有无矛盾？ | `结构连通性`、`graph_structure`、锚定比例、稀疏度 | 结构可用，且项目锚定主链清楚 |
| E. 任务代表性 Purpose Fit / Representativeness | 语料是不是服务创新创业项目？偏不偏科？两类都有吗？ | `类别均衡度`、`category_distribution`、`lifecycle_representativeness` | 类别分布较均衡，适合作为双创课程知识底座 |
| F. 方法学透明性 Methodological Soundness | 有没有承认限制？是不是无人工金标？有没有学术依据？ | `method_disclosures`、公式、参考文献、限制说明 | 方法透明，且显式承认无金标、采用弱监督代理指标 |

### 1.1 为什么不用“一个总分”讲完

因为老师问的其实不是“高不高”，而是：

1. 这批知识对不对？
2. 它全不全？
3. 它能不能回到证据？
4. 它是不是组织成了真正可用的图？
5. 它是不是服务于创新创业任务，而不是偏科语料？
6. 这套评估方法本身靠不靠谱？

所以前端和文档都必须结构化展示，而不能只放一个综合分。

### 1.2 评估总流程

```mermaid
flowchart LR
  A[原始案例/项目文档] --> B[LLM结构化抽取]
  B --> C[项目画像与证据字段]
  C --> D[Neo4j知识图谱]
  D --> E[/api/kb-stats 实时合理性]
  B --> F[scripts/evaluate_kg_quality.py 离线审计]
  F --> G[/api/kg/quality 离线报告]
  E --> H[知识图谱质量面板]
  G --> H
```

这里有一个很重要的设计思想：

- 大模型负责“抽”
- 评估模块负责“审”

也就是说，系统并不是让大模型自己给自己打分，而是尽量把评估改造成**计数、分布、覆盖率、结构指标、审计规则**的组合。

## 二、数据来源与两套评估口径

知识图谱这部分前端实际上同时消费两套数据：

| 数据源 | 接口 / 文件 | 作用 | 当前口径 |
| --- | --- | --- | --- |
| 实时知识库统计 | `/api/kb-stats` | 反映当前 Neo4j 中真实的节点、关系、分布与合理性 | 面向“现在这个库是否可用” |
| 离线质量报告 | `/api/kg/quality` -> `data/kg_quality/quality_report.json` | 反映结构化抽取与离线审计结果 | 面向“这套抽取过程是否可靠” |

因此前端看到的不是一个单一口径，而是：

- `实时图谱合理性`
- `离线抽取质量`

这两者各有侧重：

- 实时统计更偏 D、E、F 三类问题
- 离线报告更偏 A、B、C 三类问题

## 四、把大框架落到具体测评逻辑

下面按 A-F 六个大维度逐项展开。

## A. 抽取正确性 Correctness

### A.1 老师关心什么

老师真正问的是：

> 抽出来的东西是不是真的？是不是把“商业模式”抽成了“方案”，或者把“方案”抽成了“痛点”？

### A.2 系统怎么测

这一层主要对应三类指标：

| 指标 | 来源 | 逻辑 |
| --- | --- | --- |
| `抽取准确率` | `evaluate_kg_quality.py` | 随机抽样 20 个案例，检查 8 个核心画像字段是否被正确填充 |
| `语义有效性` | `graph_service.py` | 看标签边界、常见混淆、证据一致性和结构闭环 |
| `sample_audit_summary` / `audit_summary` | `graph_service.py` | 对带证据项目做分层抽样审计，看是否经得起复核 |

### A.3 当前结果

| 子指标 | 当前值 | 解释 |
| --- | ---: | --- |
| 抽取准确率 | `90.6` | 抽样 20 个案例，160 项字段检查，145 项通过 |
| 语义有效性 | `0.7202` | 当前六维里最弱的一项 |
| 审计总体通过率 | `0.9954` | 108 个项目、432 次检查、430 次通过 |

### A.4 如何解释

这一层最值得写进说明书的判断不是“整体不错”，而是：

1. 系统已经通过了“字段非空”阶段。
   现在真正的难点已经转向“贴标签贴得准不准”。

2. `语义有效性`之所以是最低项，不代表系统不行，反而说明评估变严格了。
   它开始问更难的问题，例如：
   - 商业模式是不是误抽成方案
   - 方案是不是误抽成痛点
   - 标签有没有越界

3. 所以 A 维度的结论应写成：

> 抽取正确性整体较强，但当前最需要持续优化的是“标签边界的细粒度语义判别”，也就是高阶语义正确性，而不是基础字段提取能力。

---

## B. 抽取完整性 Completeness

### B.1 老师关心什么

老师想知道：

> 该有的都有吗？高频概念是不是真的高频？低频是不是合理低频？

### B.2 系统怎么测

这一层对应：

| 指标 | 来源 | 逻辑 |
| --- | --- | --- |
| `知识覆盖率` | `evaluate_kg_quality.py` | 本体定义的概念是否都在案例库中出现 |
| `dimension_coverage` | `graph_service.py` | 9 个核心维度是否都有实体 |
| `dimension_missing_rate` | `graph_service.py` | 每个维度在项目层面的缺失率 |
| `adjusted_missing_control` | `graph_service.py` | 对合理低频维度做缺失惩罚调整 |
| `low_frequency_dimensions` | `graph_service.py` | 显式声明哪些维度低频是合理的 |

### B.3 当前结果

| 子指标 | 当前值 |
| --- | ---: |
| 知识覆盖率 | `100.0 / A+` |
| 维度覆盖率 | `1.0000` |
| 调整后缺失控制 | `0.9297` |

当前最值得关注的缺失率：

| 维度 | 缺失率 | 是否合理低频 |
| --- | ---: | --- |
| 风控 | `0.2883` | 是 |
| 执行步骤 | `0.1802` | 是 |
| 商业模式 | `0.1712` | 否 |
| 市场 | `0.0450` | 否 |
| 证据 | `0.0270` | 否 |
| 目标用户 | `0.0270` | 否 |

### B.4 如何解释

这个维度最重要的不是“谁缺得多”，而是“为什么缺”：

- `风控`和`执行步骤`被后端明确标注为合理低频：
  - `执行步骤`通常只在方案较具体、里程碑较明确时出现
  - `风控`通常集中在高监管、高安全或实施复杂度较高的项目

- `商业模式`则不同。
  它是高频通用维度，却仍有 `0.1712` 的缺失率，因此它才是当前最值得继续补强的通用维度。

所以 B 维度的结论应写成：

> 知识图谱在“有没有抽到”这件事上总体已经完整，但在通用高频维度里，商业模式仍是最值得继续补强的一环；而执行步骤与风控属于合理低频，不应被误判为系统失真。

## C. 证据可追溯 Traceability / Provenance

### C.1 老师关心什么

老师最直接的问题是：

> 每条抽出来的东西能不能回到原文？有没有 quote？能不能定位？

### C.2 系统怎么测

这一层主要对应：

| 指标 | 逻辑 |
| --- | --- |
| `traceability` | 有证据引用的案例占比 + quote 非空率 |
| `traceability_rate` | 含 quote/source_unit 的实体比例 |
| `project_traceable_coverage` | 至少有一条可追溯证据的项目比例 |
| `locatable_evidence` | 同时含 `quote` 与 `source_unit` 的项目比例 |
| `audit_closure` | 可定位 + 多证据 + 多维标签 三项同时满足的比例 |

### C.3 当前结果

| 子指标 | 当前值 |
| --- | ---: |
| 证据可溯源性 | `89.4 / A` |
| 实体可追溯率 | `0.9754` |
| 项目可追溯覆盖 | `0.9730` |
| 证据定位完备率 | `1.0` |
| 审计闭环率 | `0.9907` |

### C.4 如何解释

这是当前知识图谱最稳定的优势之一。

因为系统不仅在说“有证据”，还在进一步要求：

- 证据要有 `quote`
- 证据要有 `source_unit`
- 最好不是单证据
- 最好还能同时覆盖多个分析维度

所以 C 维度的结论可以直接写成：

> 当前知识图谱已经具备很强的可溯源性，不仅能够回到原文，还能形成“可定位、可复核、多证据、多维度”的审计闭环。

## D. 结构合理性 Structural Coherence

### D.1 老师关心什么

老师问的是：

> 这是不是一个真正可用的图？有没有主链？是不是一堆散点？

### D.2 系统怎么测

结构合理性主要看：

| 指标 | 当前值 | 含义 |
| --- | ---: | --- |
| 总节点 | `3450` | 图规模 |
| 总关系 | `9723` | 图连接规模 |
| 平均度 | `5.64` | 节点平均连接数 |
| 图密度 | `0.001634` | 图整体稠密度 |
| 项目锚定关系占比 | `0.5043` | 多少关系仍然锚回 `Project` |
| 孤立节点数 | `50` | 完全断开的节点数 |
| 稀疏节点比例 | `0.7351` | 度小于等于 2 的节点比例 |

### D.3 如何解释

如果只看`稀疏节点比例`，很容易误判这图“太散”。  
但创业知识图谱天然包含大量：

- 领域实体
- 方法概念
- 证据片段
- 单次命中的风险节点

它们并不一定都要高连接。

真正关键的是：

1. 是否有稳定锚点  
   当前 `project_anchor_ratio = 0.5043`，说明一半以上的主关系仍围绕项目主体展开。

2. 是否形成了“项目 -> 维度 -> 证据/规则/评分”的主链  
   从关系类型上看，`HAS_EVIDENCE`、`HITS_RULE`、`EVALUATED_BY`、`SUPPORTED_BY` 都很活跃，说明主链条是存在的。

所以 D 维度的结论不应写成“越密越好”，而应写成：

> 当前知识图谱的结构是“项目锚定 + 多维展开”的可用结构，虽包含大量低度领域节点，但主链清晰、锚点稳定，并非无组织散点。

## F. 方法学透明性 Methodological Soundness

### F.1 老师关心什么

老师不是只看结果，还会追问：

> 你这套方法承不承认局限？有没有人工金标？是不是只让模型自己说自己对？

### F.2 系统怎么做

这一层主要体现在：

- `method_disclosures.references`：显式列出方法学参考文献
- `method_disclosures.limitations`：显式列出限制
- `score_formula`：把综合分公式公开
- `chip_types`：明确哪些是硬指标、代理指标、抽样指标

### F.3 当前公开披露的限制

后端已明确承认至少三点：

1. 没有人工金标准  
   因此“正确性”采用弱监督代理法，而不是人工逐条标注校验。

2. 核心概念清单存在关键词匹配局限  
   可能低估同义词未覆盖的命中。

3. 行业到阶段的映射是静态启发式  
   个别案例可能被错分，但在整体统计上仍有意义。

### F.4 如何解释

这很关键，因为它说明系统不是在“伪装绝对正确”，而是在：

- 主动公开方法
- 主动公开限制
- 主动说明哪些值是硬统计，哪些值只是代理判断

所以 F 维度的结论应写成：

> 这套知识图谱评估不是黑箱分数，而是一套带有公式、文献依据、局限说明和指标类型标注的透明评估体系。

## 六、与大模型的关系

### 6.1 大模型负责哪里

大模型主要负责：

- 从原始案例中抽取结构化字段
- 生成项目画像
- 绑定证据锚点

### 6.2 大模型不负责哪里

大模型**不直接决定最终质量分**。  
评估侧主要依赖：

- 计数
- 分布
- 覆盖率
- 熵
- 图结构统计
- 抽样审计
- 弱监督语义代理指标

### 6.3 一句话结论

> 本系统不是“让大模型给自己打分”，而是让大模型负责知识抽取，再由程序化统计、结构分析和审计机制完成质量评估。
