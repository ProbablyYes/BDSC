# 商业项目与公益项目的编排区分及 Prompt 设计

[返回总目录](./README.md)

---

## 一、文档定位

本文档用于说明：当系统面对不同类型的创新创业项目时，多智能体编排机制与提示词设计如何有针对性地区分**商业性项目**与**公益性项目**。

从工程实现角度看，当前系统底层仍共享同一套诊断框架，并不是两个完全独立的引擎，从系统设计总结与答辩展示角度看，我们将其抽象为一套更适合解释的机制：

> **统一底座 + 类型识别 + 编排偏置 + Prompt 差异化**

也就是说，系统先用统一规则判断“这个项目是否说得清、站得住、能推进”，再根据项目类型把分析重心偏向不同方向，从而让商业项目与公益项目在输出风格、追问方向和下一步任务上表现出明显区别。

---

## 二、核心结论先行

### 1. 一句话概括

- 对于**商业性项目**，系统重点验证的是：项目是否形成了从用户价值到商业回路的**商业闭环**。
- 对于**公益性项目**，系统重点验证的是：项目是否形成了从社会问题到持续干预的**影响力闭环**。

### 2. 两类项目的核心差异总览

| 比较维度 | 商业性项目 | 公益性项目 |
| --- | --- | --- |
| 核心目标函数 | 验证项目能否成立、增长、变现 | 验证项目能否产生社会价值并长期持续 |
| 重点判断对象 | 用户、市场、竞品、渠道、收入、财务 | 受益人群、社会问题、干预机制、影响评估、合作网络 |
| 证据优先级 | 支付意愿、转化率、市场数据、单位经济 | 覆盖人数、前后测、反馈记录、合作证明、伦理治理 |
| 风险关注点 | 伪需求、伪增长、定价失真、财务断裂 | 情怀过强、证据过弱、效果不可测、持续性不足 |
| 下一步任务类型 | 商业验证任务 | 影响验证任务 |
| 路演表达气质 | 强调成立性、竞争力、增长性 | 强调公共性、可信度、可持续性 |

### 3. 为什么必须区分

如果不做这层区分，系统很容易犯两类典型错误：

1. 用商业项目的标准去苛求公益项目，结果不断追问盈利与回本，却忽略了社会价值和公共性。
2. 用公益项目的表达习惯去放松商业项目，结果只谈愿景和意义，却没有追问付费、转化和财务逻辑。

因此，这种区分并不是机械分类，而是为了让系统对不同项目采用更合适的判断语言、证据要求和推进路径。

## 三、总体架构：统一底座，双路偏置

### 1. 统一底座

无论是商业项目还是公益项目，系统都会先经过一层通用判断，确保最基本的问题被覆盖：

- 项目对象是否清晰
- 痛点或社会问题是否具体
- 方案是否与问题对应
- 证据是否支持结论
- 执行路径是否具备可落地性

这意味着两类项目虽然目标不同，但都不能绕开“对象明确、问题成立、方案有效、证据支撑、行动清晰”这五个底层判断。

### 2. 双路偏置

在通用判断之上，系统进一步施加项目类型偏置：

- 对于**商业性项目**，系统更强调市场成立性、用户付费意愿、渠道可达性、收入模型与财务逻辑。
- 对于**公益性项目**，系统更强调受益对象识别、社会价值链条、影响力可测量性、合作网络与持续运行机制。

### 3. 设计收益

这种设计的好处在于：

- 保证底层框架统一，便于系统维护与扩展；
- 保证不同场景下输出气质明显不同，利于答辩展示；
- 保证“同一个项目”在不同阶段也可切换关注重点，例如公益项目后期也可以增加可持续运营分析，商业项目也可以增加社会价值表达分析。

## 四、项目类型识别机制

为了让系统在交互早期就进入更合适的分析模式，我们将其总结为一个三层识别机制。

### 1. 第一层：显式关键词识别

当输入中出现以下表达时，系统会优先判定为公益型或社会创新型项目：

- 公益
- 社会问题
- 乡村振兴
- 弱势群体
- 可持续
- ESG
- 社区服务
- 志愿者
- 公益基金

当输入中出现以下表达时，系统会优先判定为商业型项目：

- 商业模式
- 收费
- 定价
- 获客
- 转化
- 竞品
- 融资
- 市场规模
- 盈利
- 回本

这一层的作用，是对输入文本做项目语义方向的快速锚定。

### 2. 第二层：目标函数识别

仅靠关键词还不够，因此系统还会进一步判断用户真正想优化的目标是什么。

如果输入更关注以下问题，系统会倾向判定为商业项目：

- 用户为什么愿意付费
- 首批用户怎么来
- 收入结构怎么搭
- 竞品比较后优势在哪里
- 财务上能否跑通

如果输入更关注以下问题，系统会倾向判定为公益项目：

- 真正受影响的人是谁
- 问题是否真的存在
- 项目能带来什么社会改变
- 如何证明干预有效
- 没有持续资助时如何继续运行

也就是说，系统识别的不是“项目自称是什么”，而是“项目到底在追求什么结果”。

### 3. 第三层：证据形态识别

系统还会观察项目当前使用的是哪类证据。不同证据结构会反向强化项目类型判断。

- 商业项目更常出现：付费意愿、市场规模、转化率、CAC、LTV、复购、客单价、渠道 ROI。
- 公益项目更常出现：覆盖人数、服务频次、满意度、前后对比、案例访谈、合作单位反馈、行为改善指标。

### 4. 最终识别结论

因此，系统并不是简单根据名称分类，而是综合以下三类信号来决定后续编排偏置：

1. **关键词信号**
2. **目标函数信号**
3. **证据结构信号**

这也是我们能在答辩中强调“系统具备项目类型感知能力”的核心依据。

## 五、智能体编排如何区分两类项目

在同一套多智能体底座之上，系统会对不同项目类型施加不同的编排重心，使得分析路径产生明显区别。

### 1. 编排差异不是换智能体，而是换“主问题”

我们并不是为商业项目和公益项目各自重新开发一套智能体，而是让同一组智能体在面对不同项目时回答不同的主问题：

| 智能体 | 商业项目下的主问题 | 公益项目下的主问题 |
| --- | --- | --- |
| 项目教练 Agent | 这个项目的核心商业瓶颈是什么？ | 这个项目的核心影响力瓶颈是什么？ |
| 竞赛顾问 Agent | 评委会质疑它是否能成立、增长、变现吗？ | 评委会质疑它是否真实有效、值得推广、可持续吗？ |
| 批判 Agent | 哪些地方在商业叙事上跳步了？ | 哪些地方在社会价值叙事上空泛了？ |
| 规划 Agent | 下一步最小商业验证动作是什么？ | 下一步最小影响验证动作是什么？ |

这意味着，真正区分两类项目的，不只是标签本身，而是**智能体在本轮被要求优先回答的判断目标**。

### 2. 商业项目的核心判断链

商业项目场景下，系统默认优先检查：

> **目标用户 -> 核心痛点 -> 产品方案 -> 差异化优势 -> 商业模式 -> 获客路径 -> 财务逻辑**

这条链条的终点不是“项目有没有意义”，而是“项目能否构成一个可持续的商业闭环”。

### 3. 公益项目的核心判断链

公益项目场景下，系统默认优先检查：

> **受益对象 -> 社会问题 -> 干预机制 -> 影响指标 -> 合作网络 -> 资源支持 -> 持续运行机制**

这条链条的终点不是“项目是否感人”，而是“项目是否构成一个可信的影响力闭环与可持续闭环”。

---

## 六、商业项目的编排重点

### 1. 编排目标

商业项目的编排目标，是把一个“看上去有创意的想法”压缩成一个“具备市场成立可能的项目模型”。

### 2. 各智能体的关注重心

- **项目教练 Agent**
  重点识别市场切口、用户聚焦程度、替代方案、当前最大商业瓶颈。

- **竞赛顾问 Agent**
  重点从评委视角检查商业模式、市场规模、竞品比较、融资叙事、团队商业化路径是否成立。

- **批判 Agent**
  重点挑战“伪需求”“伪增长”“伪市场规模”“定价拍脑袋”“只谈功能不谈价值”。

- **规划 Agent**
  将建议压缩成最小商业验证动作，例如用户访谈、预约页测试、价格接受度验证、小规模渠道测试、首批客户触达表。

### 3. 系统输出的典型偏置

商业项目的回复中，更可能出现以下内容：

- 对目标客群的收缩建议
- 对支付意愿和替代方案的追问
- 对获客路径和转化漏斗的拆解
- 对收入模式、成本结构和盈亏平衡的判断
- 对下一步商业验证动作的明确要求

### 4. 本质解释

因此，商业项目的回复读起来更像是在回答：

> **“这个项目能不能成立、能不能增长、能不能跑起来？”**

---

## 七、公益项目的编排重点

### 1. 编排目标

公益项目的编排目标，是把一个“价值感很强的倡议”压缩成一个“真实有效且可持续运行的社会创新方案”。

### 2. 各智能体的关注重心

- **项目教练 Agent**
  优先识别问题定义是否具体、受益对象是否真实、干预机制是否真正作用于问题本身。

- **竞赛顾问 Agent**
  重点从社会价值表达、公共性、可推广性、影响评估和合作机制角度判断项目说服力。

- **批判 Agent**
  重点挑战“情怀过强、证据过弱”“目标人群模糊”“效果不可测量”“过度依赖单一资助”“一次性活动冒充长期项目”。

- **规划 Agent**
  优先生成影响力验证任务，例如前后测设计、样本追踪、合作学校试点、服务频次记录、反馈收集机制、治理流程补充。

### 3. 系统输出的典型偏置

公益项目的回复中，更可能出现以下内容：

- 对受益对象边界的追问
- 对社会问题证据和现场感的要求
- 对影响力指标定义与测量方式的建议
- 对合作机制、资源来源和治理方式的补充
- 对项目持续运行而非单次活动效果的判断

### 4. 本质解释

因此，公益项目的回复读起来更像是在回答：

> **“这个项目是否真的产生社会价值，并且能否持续？”**

## 八、Prompt 设计的四层差异

为了让两类项目在生成侧表现出更明显的差别，系统在 Prompt 设计上做四层差异化控制。

### 1. 角色设定不同

商业项目更适合把模型设定为：

- 创新创业项目教练
- 商业模式顾问
- 路演评审顾问

公益项目更适合把模型设定为：

- 公益创新项目教练
- 社会影响力评估顾问
- 公益路演辅导顾问

角色名称的变化不是装饰性的，而是为了让模型自动切换判断框架。

### 2. 评价标准不同

商业项目 Prompt 更强调：

- 用户价值是否成立
- 市场空间是否可信
- 收费逻辑是否成立
- 增长路径是否可执行
- 财务结构是否合理

公益项目 Prompt 更强调：

- 社会问题是否被准确定义
- 干预方案是否真正有效
- 影响结果是否可以衡量
- 资源网络是否稳定
- 项目是否具有持续运行机制

### 3. 证据要求不同

商业项目 Prompt 会优先索要：

- 用户访谈
- 转化数据
- 定价反馈
- 市场数据
- 竞品矩阵
- 财务测算

公益项目 Prompt 会优先索要：

- 受益对象原始反馈
- 前后测数据
- 覆盖人数与服务频次
- 合作机构证明
- 影响力指标
- 风险与伦理说明

### 4. 下一步任务不同

商业项目的下一步任务更像是“商业验证任务”：

- 访谈 5 位核心用户
- 做一次预约页点击测试
- 验证价格接受度
- 拆解首批用户触达路径
- 建立基础 CAC/LTV 估算表

公益项目的下一步任务更像是“影响验证任务”：

- 明确 1 类核心受益人群并完成样本访谈
- 设计 1 组前后测指标
- 建立服务记录表与反馈回收机制
- 找到 1 个合作试点场景
- 补齐资源来源与持续运行预案

## 九、两类项目的 Prompt 模板

### 1. 商业项目的 Prompt 设计

```text
你是一名创新创业项目教练，请从商业成立性的角度分析该项目。
不要直接代写计划书，而是重点判断：
1. 目标用户是否足够清晰；
2. 痛点是否有真实证据；
3. 解决方案相对现有替代方案是否有可感知改进；
4. 商业模式是否成立，包括谁付费、为什么付费、如何持续付费；
5. 获客渠道是否真实可达；
6. 财务逻辑是否具备初步可行性。

请输出：
- 当前最关键的商业瓶颈
- 3个苏格拉底式追问
- 1个下一步唯一任务
- 该任务的验收标准
```

**设计意图说明：**  
这一类 Prompt 的目标，不是生成一段表面流畅的商业计划书，而是推动模型围绕“商业闭环是否成立”做判断。因此它更强调聚焦、验证、转化和数字化。

### 2. 公益项目的 Prompt 设计

```text
你是一名公益创新创业项目教练，请从“社会影响力”和“持续运行机制”两个维度分析该项目。
不要只评价项目是否有意义，而要重点判断：
1. 目标受益人群是否清晰；
2. 社会问题是否被具体定义；
3. 干预方案是否真正作用于问题；
4. 影响结果能否被衡量；
5. 项目是否具备资源支持、合作网络与持续运行机制；
6. 是否存在伦理、隐私或治理风险。

请输出：
- 当前最关键的瓶颈
- 3个启发式追问
- 1个下一步唯一任务
- 影响力验证的验收标准
```

**设计意图说明：**  
这一类 Prompt 的重点，是防止模型把公益项目简单理解成“有温度的故事”或“只要有意义就值得做”。因此它会主动把项目压实到受益对象、干预效果、影响力指标和持续机制上。

## 十、完整场景测试：商业项目

> 这一部分用于回应“老师要求完成场景测试”的展示需求。我们不仅说明系统怎么区分，还展示系统在完整输入场景下会如何工作。

### 1. 测试场景设定

假设输入项目为：

> “我们想做一个面向高校大学生的 AI 求职模拟面试平台，通过大模型生成真实问题、点评表达逻辑和简历匹配度。目标是在秋招前帮助学生提升面试通过率。我们计划先从考研转就业人群切入，后续通过校园大使、求职社群和 B 站内容获客，并尝试会员订阅和按次训练收费。”

### 2. 系统识别结果

在这一输入下，系统会优先命中以下商业信号：

- 出现“收费”“获客”“会员订阅”“按次训练”等商业模式关键词
- 目标函数偏向“用户为什么付费、渠道如何增长、收入如何成立”
- 证据需求自然会转向支付意愿、渠道测试、转化率和客单价

因此，系统会将该场景归入**商业性项目分析模式**。

### 3. 预期编排流程

| 阶段 | 触发重点 | 主要任务 |
| --- | --- | --- |
| 阶段 1 | 项目教练 Agent | 识别目标用户是否收敛，判断核心痛点是否真实 |
| 阶段 2 | 竞赛顾问 Agent | 检查商业模式、竞品差异、市场规模与融资叙事 |
| 阶段 3 | 批判 Agent | 反驳“人人都需要”“大市场自然会来”的跳步逻辑 |
| 阶段 4 | 规划 Agent | 给出最小商业验证任务和验收标准 |

### 4. 预期系统追问

系统在该场景下更可能发出以下追问：

1. 你的首批核心用户到底是“所有求职大学生”，还是“考研转就业且缺少线下面试资源的人”？
2. 用户当前是怎么解决这个问题的，你的方案比现有替代方式强在哪里？
3. 如果不考虑“AI”这个标签，你给用户带来的可感知价值是什么？
4. 用户愿意持续付费，还是只愿意在秋招前短期购买一次？
5. 校园大使和求职社群的转化路径各是什么，哪个才是首轮重点渠道？

### 5. 预期下一步任务

在商业模式下，系统更可能给出的唯一任务是：

> **完成一轮最小支付意愿与渠道验证**

建议验收标准包括：

- 访谈 5 至 8 位目标用户
- 验证至少 2 个高频痛点是否真实存在
- 测试 1 个价格区间的接受度
- 记录 1 条真实可执行的首批触达路径

### 6. 测试结论

该场景下，系统不会把重点放在“项目很有帮助、社会价值不错”这种泛化表述上，而是会持续把问题压向：

- 用户是否足够聚焦
- 用户为什么愿意为此付费
- 渠道是否可达
- 收入模型是否站得住

这体现出商业项目场景下，系统编排与 Prompt 的核心目标是**验证商业闭环**。

## 十一、完整场景测试：公益项目

### 1. 测试场景设定

假设输入项目为：

> “我们希望为乡村留守儿童设计一个 AI 阅读陪伴与心理关怀平台，通过语音互动、分层阅读内容和教师端反馈功能，帮助孩子提升阅读兴趣与情绪表达能力。项目计划先与县域学校合作试点，再引入公益基金和企业 CSR 资源支持，逐步扩大服务范围。”

### 2. 系统识别结果

在这一输入下，系统会优先命中以下公益信号：

- 出现“留守儿童”“心理关怀”“合作试点”“公益基金”“企业 CSR”等社会创新关键词
- 目标函数偏向“谁真正受益、干预是否有效、项目如何持续”
- 证据需求自然会转向覆盖人数、前后测、教师反馈、合作证明和治理机制

因此，系统会将该场景归入**公益性项目分析模式**。

### 3. 预期编排流程

| 阶段 | 触发重点 | 主要任务 |
| --- | --- | --- |
| 阶段 1 | 项目教练 Agent | 判断受益对象是否清晰、问题定义是否具体 |
| 阶段 2 | 竞赛顾问 Agent | 检查社会价值表达、合作机制与可推广性 |
| 阶段 3 | 批判 Agent | 挑战“只讲情怀不讲证据”“只讲活动不讲机制”的问题 |
| 阶段 4 | 规划 Agent | 给出最小影响验证任务和持续运行补强方案 |

### 4. 预期系统追问

系统在该场景下更可能发出以下追问：

1. 你服务的核心对象到底是全部乡村儿童，还是“缺少监护陪伴且阅读资源不足的留守儿童”？
2. 你要解决的核心问题是阅读资源缺失、陪伴缺失，还是情绪表达困难？三者中当前最核心的是哪一个？
3. 你如何证明平台介入后孩子的阅读兴趣或表达能力真的发生了变化？
4. 如果没有持续基金支持，学校和团队是否还能维持项目运行？
5. 教师、家长、志愿者在项目中的角色分工是否已经明确？

### 5. 预期下一步任务

在公益模式下，系统更可能给出的唯一任务是：

> **完成一轮小规模影响力验证与试点机制设计**

建议验收标准包括：

- 明确 1 类核心受益人群
- 设计 1 组前后测指标
- 完成 1 个试点学校或社区场景对接
- 建立基础服务记录与反馈回收机制
- 补齐资源来源与持续运行说明

### 6. 测试结论

该场景下，系统不会把重点放在“市场很大”“未来可以收费”这类商业语汇上，而是会持续把问题压向：

- 受益对象是否明确
- 社会问题是否真实存在
- 干预效果是否可测
- 合作机制是否稳定
- 项目是否具备持续运行条件

这体现出公益项目场景下，系统编排与 Prompt 的核心目标是**验证影响力闭环与持续机制**。

## 十二、为什么这两组场景测试能够体现系统差异

从两组测试可以看到，系统的差异并不只是“商业项目多问钱，公益项目多问意义”这么简单，而是体现在以下四个层面：

### 1. 识别层不同

系统先判断项目到底在追求“商业成立”还是“社会改变”，从源头决定后续分析方向。

### 2. 编排层不同

同样是项目教练、竞赛顾问、批判 Agent 和规划 Agent，但它们被要求优先回答的问题并不相同。

### 3. Prompt 层不同

商业项目 Prompt 更强调用户、市场、转化和财务；公益项目 Prompt 更强调受益对象、影响评估、合作网络和持续机制。

### 4. 输出层不同

商业项目最终输出更像“商业验证建议”，公益项目最终输出更像“影响验证与机制补强建议”。

这也是本文档最想突出的核心区别：  
**我们不是简单地把项目贴上标签，而是让整条分析链在不同项目类型下发生重心迁移。**

---

## 十三、可直接写入说明书的正式表述

在系统设计上，我们将商业性项目与公益性项目统一纳入同一套多智能体诊断框架中，但在编排层与 Prompt 层引入了显式的类型偏置机制。系统首先通过关键词、目标函数和证据形态识别项目类型，再对智能体的关注重点、批判方向、追问方式与下一步任务进行差异化调度。

对于商业性项目，系统更强调“用户价值 - 商业模式 - 增长路径 - 财务逻辑”的商业闭环；对于公益性项目，系统更强调“社会问题 - 干预方案 - 影响评估 - 持续运行”的影响力闭环。基于这一机制，系统在完整场景测试中能够对两类项目给出明显不同的分析路径与任务建议，体现出其在创新创业教育场景中的类型感知能力、编排能力与提示词适配能力。
